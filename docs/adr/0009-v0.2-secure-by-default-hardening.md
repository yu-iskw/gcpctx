# ADR 0009: v0.2 secure-by-default hardening

- **Status:** Accepted
- **Date:** 2026-06-23
- **Deciders:** Maintainers

## Context

Security review identified boundary-hardening gaps: project identity split-brain via `env.CLOUDSDK_CORE_PROJECT`, non-atomic state writes, ambient PATH trust for `gcloud`, Windows permission bypass, and hook mode exporting credentials without ADC readiness.

## Decision

Ship **v0.2** with:

1. **Immutable project identity** — remove `CLOUDSDK_CORE_PROJECT` from profile `env` allowlist; always export `profile.project`.
2. **Hardened state store** — `secure_atomic_write_text`, advisory locks, symlink rejection for managed paths.
3. **gcloud trust** — validate resolved binary path; bind path + SHA-256 in approval schema v2.
4. **Policy engine** — optional `policy.toml` for allowlists, TTL, strict hook ADC requirement.
5. **Hook readiness** — `approved_not_initialized` state deactivates shell when ADC missing.
6. **`doctor --strict`** — machine-readable compliance gate with IAM impersonation probe.
7. **Audit JSONL** — append-only security events without credential material.
8. **Windows fail-closed** — CLI rejects `win32` until ACL support exists.
9. **Release provenance** — SBOM + build attestation on GitHub Release; PyPI Trusted Publishing.

## Consequences

- Breaking: configs with `env.CLOUDSDK_CORE_PROJECT` fail validation.
- Breaking: v1 approvals without gcloud binding may require re-approval in strict mode.
- Breaking: Windows users cannot run gcpctx until v0.3+ ACL work.
- Non-breaking: default policy absent retains permissive behavior except P0 invariants.

## Alternatives considered

- **Brokered token service** — deferred; too much infrastructure for a local CLI.
- **Windows warn-only** — rejected; false security claims for credential tooling.
